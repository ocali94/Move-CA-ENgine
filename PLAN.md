# Move CA Engine — Upgrade Plan (Phase 2)

Decision rule applied: rebuild only where fixing costs more than rewriting. Result: **no module is rebuilt from scratch**; auth is replaced wholesale (per spec), everything else is refactored in place. The UI layer, types, and localStorage state machine survive.

## Order of work (matches Phase 3 priority)

### 1. Simplify auth (replace NextAuth)
- Delete: `src/lib/auth.ts`, `src/app/api/auth/[...nextauth]/`, `src/types/next-auth.d.ts`, all four `/api/drive/*` routes, the Google Picker code in `proposal-studio-client.tsx`, `SessionProvider` in providers, `signIn/signOut/useSession` usages.
- New: `POST /api/auth/login` checks `ACCESS_CODE`, sets an httpOnly SHA-256-of-code cookie (`move_ca_access`); `POST /api/auth/logout` clears it. `src/proxy.ts` (Next 16 convention, named `proxy` export, Node runtime) gates `/dashboard/*`, `/setup`, and `/api/*` (except `/api/auth/login` and `/api/health`): pages redirect to `/login`, APIs get 401 JSON. `src/lib/server-auth.ts` shrinks to a cookie check used by the proxy and as defense-in-depth in layouts.
- If `ACCESS_CODE` is unset, the login page says so explicitly (dev convenience: any non-empty code is rejected with a clear "ACCESS_CODE not configured" message — no silent bypass).
- Remove deps: `next-auth`, `mammoth`. Keep everything else (all used).
- Paste-text reference import stays (it becomes `/api/references/import`); `content/` is the only reference library.

### 2. Server-side website fetch
- New `POST /api/fetch-website`: zod-validated URL, 10s `AbortController` timeout, follows redirects, rejects private/non-http(s) hosts, strips `<script>/<style>/<nav>/<footer>`, extracts title + meta description + headings + body text (regex-based; no new deps), caps at ~8,000 chars. Returns `{ok, title, description, text}` or `{ok:false, error}` — always 200-shaped so a failed fetch can't throw a module off course.
- Lead Qualifier: if the website field (or pasted notes) looks like a URL, auto-fetch before analyze; on failure show amber "Couldn't reach the site — scoring on pasted text only" notice and continue.
- Call Prep: extract first URL from intake/notes/website field, same auto-fetch + same visible-failure behavior.

### 3. LLM pipeline + status visibility (prerequisite for everything generated)
- Wire the existing (currently dead) prompts into real server-side generation: each module route tries `generateWithLLM` (JSON-mode prompts, zod-parsed) and **falls back to the existing deterministic workflow** on missing key, timeout, or parse failure. The heuristics finally become what the README claimed they were: the fallback.
- Every generation response carries `generation: {mode: "llm" | "fallback", provider?, model?, reason?}`.
- `GET /api/llm/status` reports configured + last-call-result (module-level in-memory record). Header badge: green "Live AI" / amber "Fallback mode", polled on load and updated after each generate call. Every output panel gets a small footer: "Generated by Claude (claude-sonnet-4-5)" or "Local fallback logic — no LLM key configured / last call failed".

### 4. Market Signals polish
- Replace dead Stooq with Yahoo Finance chart API (keyless, verified working) for XLY/XLP monthly closes.
- `Promise.all` → `Promise.allSettled`: one dead source can no longer kill the dashboard. Partial results get per-signal `dataMode`; cache merges last-good per signal.
- Derive each card's "What it means" from actual direction (rule per series × trend, written for zero-economics readers). Cards: value, trend arrow, sparkline (all exist), plus the derived line.
- Demand Pulse: keep gauge + math; add trend arrow + one-sentence interpretation (exists as `summary`, tighten).
- Campaign Signal: restructure to exactly three blocks — "What changed this week" (from real deltas), "What it means for $1M–$50M DTC brands", "What Move should campaign on now" (2–3 angles). LLM-written from the fetched numbers when live, deterministic from the same numbers otherwise. Only fetched values may appear; cached/demo clearly labeled (badge exists, keep).

### 5. Proposal Studio rule enforcement
- Restructure section templates to the format the overview + spec describe (the repo's `move-proposal-rules.md` is a stub; the spec's enumeration is treated as the guide's hard rules — noted as a discrepancy in AUDIT.md):
  1. Cover and Context (5–8 specific pain bullets)
  2. Engagement Options (package table: Name, Cost, Hours, Duration, Move Roles; Option 3 conditional)
  3. Transitional Timeline (table)
  4. Recommended Path (tied to extracted pains)
  5. Service Levels (exactly 3 columns: KPI, Commitment, Metrics; 3–5 rows)
  6. Scope Pillars (each pillar: Objective, Approach, Expected Outcome)
  7. Investment, Assumptions, and Next Steps
- `src/lib/proposal-rules.ts`: pure validators per section returning `{passed, issues[]}` — run after every generate/revise (LLM or fallback), shown in the UI as a checklist; blocking approval is not silent: failures are listed, the human decides.
- Official Move role names: not defined anywhere in the repo — encoded as a constant list derived from `move-service-paths.md` service language (documented in README as the place to edit). Validator flags any other role name.
- Option 3 decision tree (encoded, documented): Option 3 (Fractional Supply Chain Support) renders only when extracted facts justify ongoing support — ≥3 distinct pain categories, or a leadership/bandwidth pain, or explicit recurring-need signals. Otherwise two options only.
- Em dashes: auto-replaced before display everywhere (sections, email, exports).
- Pricing/duration consistency: extract `$amounts` and durations from Sections 2 and 7 (and timeline weeks in 3); mismatches flagged in UI for human review, never silently corrected. Pricing stays TBD unless explicitly confirmed (per `move-pricing-benchmarks.md`).
- Keep approve/lock/unlock-invalidation flow untouched. Fix HTML export to render markdown tables.
